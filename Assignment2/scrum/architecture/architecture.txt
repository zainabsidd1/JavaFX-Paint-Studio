# Please discuss your architectural decisions below.

High Level Architecture:

	MVC
	To keep the code simple, a partial MVC was implemented.
	
	Model
	The Model captures enough information to re-draw the sketch of
	the user.
	
	View+Controller
	There is a View class as well as a few, what we call, view components.
	The View class containing the BorderPane as well as containing all of its components.
	View components are meant to be contained in the main View.
	Examples of view components are PaintPanel and ShapeChooserPanel.
	
	The View class, and view components all implements their own controller.
	For example, the PaintPanel listens to its own MouseEvents.
	This is instead of separating out the Controller from the View. This choice
	was made to simplify the code, yet still keep it modular and clear.
	
	Each view component can see the containing View as well as the Model
	as appropriate. This means that each view component can speak to the
	other view components. An example of this is when the ShapeChooserPanel
	buttons are clicked, resulting in a mode change in the PaintPanel.

	Strategy:
	Originally, all mouse-handling logic for drawing shapes
	(circles, rectangles, squiggles, etc.) was crammed inside the
	PaintPanel class under one large switch(mode) statement.
    Each case handled how to:
        - respond to mouse presses, drags, and releases,
        - preview the shape being drawn, and
        - update the model once the shape was finished.
    This made the class long, hard to read, and violated the Open/Closed
    Principle — every time we added a new tool or shape, we had to edit
    the same PaintPanel file and extend the switch block. The code was
    tightly coupled and not modular.
    We introduced the Strategy pattern by creating a ToolStrategy
    interface that defines how all drawing tools behave:
    Each shape (e.g., Circle, Rectangle, Squiggle) now has its own concrete
    strategy class:
        CircleStrategy
        RectangleStrategy
        SquiggleStrategy
    Each of these classes encapsulates:
        how that shape responds to mouse events, and
        how it draws its live preview while being created.
        For example, CircleStrategy handles calculating the radius while
        dragging, and RectangleStrategy handles the second corner. The
        PaintPanel no longer contains any shape-specific logic — instead,
        it delegates all event handling to the current strategy which
        improves readability and makes the class easier to extend.

	Factory:
	Originally, the shape buttons were hard-wired to modes and PaintPanel
	logic, so adding a new tool meant editing multiple places (button
	wiring, mode switch, and event handling). We introduced a Factory
	model by registering each tool as a ToolDescriptor (name, icon path,
	ToolStrategy class). When a button is clicked, the panel instantiates
	the correct strategy via the descriptor (reflection) and sets it on
	the PaintPanel. This removes shape-specific conditionals from the UI,
	centralizes tool metadata in one list, and supports adding tools by
	appending a single descriptor, and better adherence to the Open/Closed
	Principle. It also keeps MVC cleaner: the chooser creates tools; the
	panel uses them and the model stays uninvolved.

	Builder:
	Observer/Observable:
    Originally, the `PaintModel` class extended Java’s built-in `Observable`
    and relied on the deprecated `notifyObservers()` and `setChanged()`
    methods to update the view. This approach made the model tightly coupled
    to Java’s outdated observer implementation and less flexible for expansion.
    We refactored it using a custom **Observer design pattern**, introducing
    a simple `PaintModelListener` interface with a single `modelChanged()`
    method. The `PaintModel` now maintains a list of listeners and notifies
    them directly whenever its internal state changes (such as when a shape
    is added or cleared). The `PaintPanel` implements this listener and
    updates its canvas whenever the model changes. This refactor made the
    communication between model and view explicit, modular, and easier to
    maintain. The model now focuses purely on data management, while the
    panel handles rendering, resulting in a cleaner MVC-aligned architecture.
